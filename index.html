1. O Desafio (Introdução)

Cenário: Você acaba de ser contratado como Engenheiro de Dados em uma EdTech. A equipe de arquitetura desenhou o Diagrama Entidade-Relacionamento (DER) para o novo sistema de gestão escolar.

Sua missão é acessar o Microsoft SQL Server Management Studio (SSMS) e transformar esse diagrama em um banco de dados real, operante e seguro, pronto para ser conectado à aplicação web.

Atenção à Arquitetura: Em bancos de dados relacionais, a ordem de criação importa. Não podemos criar o "telhado" antes das "paredes". Analise as chaves estrangeiras para definir por onde começar.

2. O que é SQL?

A SQL (Structured Query Language) é a linguagem universal para bancos de dados relacionais. Ela é dividida em subconjuntos vitais:

  • DDL (Data Definition Language): Define estruturas. Ex: CREATE, ALTER, DROP. (Foco deste laboratório)
  • DML (Data Manipulation Language): Manipula dados. Ex: INSERT, UPDATE, DELETE.
  • DQL (Data Query Language): Consulta dados. Ex: SELECT.
  • DCL (Data Control Language): Controla acessos. Ex: GRANT, REVOKE.
  • DTL (Data Transaction Language): Gerencia transações. Ex: COMMIT, ROLLBACK.

4. Implementação do DER (Projeto Principal)

Passo 1: Criação do Banco Oficial

SQL - Setup
CREATE DATABASE GestaoEscolar;
GO
USE GestaoEscolar;
GO

Passo 2: Entidades Fortes (Sem Dependências)

SQL - Tabelas Base
CREATE TABLE ALUNO (
    id INT IDENTITY(1,1) CONSTRAINT PK_ALUNO PRIMARY KEY,
    matricula NVARCHAR(20) CONSTRAINT UQ_ALUNO_MATRICULA UNIQUE NOT NULL,
    nome NVARCHAR(100) NOT NULL,
    data_nascimento DATE NOT NULL
);
GO

CREATE TABLE DOCENTE (
    id INT IDENTITY(1,1) CONSTRAINT PK_DOCENTE PRIMARY KEY,
    registro NVARCHAR(20) CONSTRAINT UQ_DOCENTE_REGISTRO UNIQUE NOT NULL,
    nome NVARCHAR(100) NOT NULL,
    especialidade NVARCHAR(100) NOT NULL
);
GO

CREATE TABLE CURSO (
    id INT IDENTITY(1,1) CONSTRAINT PK_CURSO PRIMARY KEY,
    nome NVARCHAR(100) NOT NULL,
    descricao NVARCHAR(MAX) NULL,
    carga_horaria INT NOT NULL
);
GO

Passo 3: Entidade de Relacionamento (1:N)

SQL - Dependência Simples
CREATE TABLE TURMA (
    id INT IDENTITY(1,1) CONSTRAINT PK_TURMA PRIMARY KEY,
    ano_semestre NVARCHAR(10) NOT NULL,
    curso_id INT NOT NULL,
    docente_id INT NOT NULL,
    CONSTRAINT FK_TURMA_CURSO FOREIGN KEY (curso_id) REFERENCES CURSO(id),
    CONSTRAINT FK_TURMA_DOCENTE FOREIGN KEY (docente_id) REFERENCES DOCENTE(id)
);
GO

Passo 4: Tabela Associativa (Resolução N:M)

SQL - Dependência Complexa
CREATE TABLE ALUNO_TURMA (
    aluno_id INT NOT NULL,
    turma_id INT NOT NULL,
    data_matricula DATE CONSTRAINT DF_ALUNO_TURMA_DATA DEFAULT GETDATE(),
    nota DECIMAL(5,2) NULL, 
    faltas INT CONSTRAINT DF_ALUNO_TURMA_FALTAS DEFAULT 0,
    CONSTRAINT PK_ALUNO_TURMA PRIMARY KEY (aluno_id, turma_id),
    CONSTRAINT FK_ALUNO_TURMA_ALUNO FOREIGN KEY (aluno_id) REFERENCES ALUNO(id),
    CONSTRAINT FK_ALUNO_TURMA_TURMA FOREIGN KEY (turma_id) REFERENCES TURMA(id)
);
GO

Passo 5: Otimização de Performance

SQL - Índices
CREATE NONCLUSTERED INDEX IX_TURMA_CURSO_ID ON TURMA(curso_id);
GO
CREATE NONCLUSTERED INDEX IX_TURMA_DOCENTE_ID ON TURMA(docente_id);
GO
CREATE NONCLUSTERED INDEX IX_ALUNO_TURMA_TURMA_ID ON ALUNO_TURMA(turma_id);
GO

5. Retrospectiva (Q&A)

Porque a tabela ALUNO_TURMA possui dependências físicas (Foreign Keys) que apontam para ALUNO e TURMA. O SGBD bloqueia a criação para garantir a integridade referencial.

Utilizamos a DDL (Data Definition Language) com os comandos CREATE e DROP para estruturar o armazenamento físico.
